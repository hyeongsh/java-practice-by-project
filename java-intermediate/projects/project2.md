# 자바 중급 개념 v2 (객체 설계/컬렉션/예외처리/파일 저장)

## 목표
- 도메인 객체를 역할 중심으로 분리해서 설계하기
- `List`, `Map` 등 컬렉션을 상황에 맞게 선택해서 사용하기
- 비즈니스 규칙 위반을 예외로 표현하고 처리 흐름 만들기
- 파일 저장/불러오기(영속화)로 프로그램 상태 유지하기
- 테스트로 핵심 대출/반납 규칙 검증하기

## 프로젝트 주제
`도서관 대출 관리 시스템 (콘솔)`

### 시나리오
- 도서관에는 여러 권의 책이 있고, 회원은 책을 대출/반납할 수 있다.
- 같은 책은 동시에 한 명만 대출할 수 있다.
- 회원은 최대 대출 가능 권수를 넘길 수 없다.
- 프로그램을 종료했다가 다시 실행해도 데이터(책/회원/대출 상태)가 유지되어야 한다.

## 요구사항

### 1. 도메인 객체 설계
아래 객체를 기본으로 설계한다. (필요하면 추가 가능)

- `Book`
- `Member`
- `Loan` (대출 기록)
- `Library` 또는 `LibraryService` (핵심 비즈니스 로직)

#### `Book` 필수 속성 예시
- `id` (도서 식별자, 중복 불가)
- `title`
- `author`
- `loaned` 또는 현재 대출 상태를 표현할 필드

#### `Member` 필수 속성 예시
- `id` (회원 식별자, 중복 불가)
- `name`
- `maxLoanCount` (기본값 지정 가능)

#### `Loan` 필수 속성 예시
- `bookId`
- `memberId`
- `loanDate`
- `dueDate`
- `returned` 또는 `returnDate`

#### 조건
- 생성 시점에 유효하지 않은 상태를 최대한 막는다. (예: 빈 문자열 id 금지)
- 상태 변경 메서드는 의미가 드러나게 작성한다. (예: `markReturned(...)`)
- `toString()`은 콘솔 출력 확인에 도움이 되도록 작성한다.
- 식별자 기반 동등성(`equals/hashCode`)이 필요한 객체는 명확한 기준을 정해 구현한다.

### 2. 핵심 기능 구현
최소 아래 기능을 구현한다.

#### 도서 관리
- 도서 등록
- 도서 목록 조회
- 도서 단건 조회 (`id` 기준)

#### 회원 관리
- 회원 등록
- 회원 목록 조회
- 회원 단건 조회 (`id` 기준)

#### 대출/반납 관리
- 도서 대출
- 도서 반납
- 현재 대출 중 목록 조회
- 회원별 대출 목록 조회

#### 조건
- 대출 시 `bookId`, `memberId`가 모두 존재해야 한다.
- 이미 대출 중인 책은 다시 대출할 수 없다.
- 회원의 현재 대출 권수가 `maxLoanCount`를 초과하면 안 된다.
- 반납 시 현재 대출 중인 기록이 있어야 한다.
- 반납 완료 후 해당 책은 다시 대출 가능 상태가 되어야 한다.

### 3. 예외 처리 설계
비즈니스 규칙 위반 상황을 예외로 처리한다.

예시 (직접 이름 정해도 됨):
- `BookNotFoundException`
- `MemberNotFoundException`
- `AlreadyLoanedException`
- `LoanLimitExceededException`
- `LoanNotFoundException`

#### 조건
- `RuntimeException` 기반 사용자 정의 예외를 최소 3개 이상 만든다.
- 예외 메시지는 원인 파악이 가능하게 작성한다. (예: 어떤 `bookId`에서 실패했는지)
- 콘솔 UI 계층(입출력)과 비즈니스 계층(서비스)의 책임을 분리한다.
  - 서비스: 예외를 던짐
  - UI: 예외를 잡아 사용자에게 메시지 출력

### 4. 컬렉션 사용 요구사항
도메인 저장소(메모리)는 컬렉션으로 구현한다.

권장 예시:
- `Map<String, Book>`: 도서 조회 최적화
- `Map<String, Member>`: 회원 조회 최적화
- `List<Loan>`: 대출 이력 저장

#### 조건
- 왜 `List` 대신 `Map`을 선택했는지(또는 반대로) 한 줄 주석/기록으로 남긴다.
- 외부에서 내부 컬렉션을 직접 수정할 수 없게 한다. (복사본/읽기 전용 뷰/DTO 등)
- 검색 로직이 중복되면 메서드로 추출한다.

### 5. 파일 저장/불러오기 (영속화)
프로그램 종료 후에도 상태가 유지되도록 파일 저장 기능을 구현한다.

구현 방식은 아래 중 하나 선택:
1. 텍스트 파일(CSV/커스텀 포맷)
2. JSON (라이브러리 사용 가능하면 사용, 어려우면 텍스트 우선)

#### 최소 저장 대상
- 도서 목록
- 회원 목록
- 대출 기록(반납 여부 포함)

#### 최소 기능
- 프로그램 시작 시 파일 로드 (`load`)
- 변경 후 저장 (`save`) 또는 종료 시 일괄 저장

#### 조건
- 파일이 없으면 예외로 종료하지 말고 빈 데이터로 시작한다.
- 파일 포맷 오류 시 처리 전략을 정한다.
  - 예: 로그 출력 후 빈 데이터로 시작 / 예외 재던지기
- 저장/로드 책임을 서비스와 분리한다. (예: `LibraryRepository`, `FileStorage`)

### 6. 콘솔 UI (메뉴 기반)
메뉴 기반으로 기능을 사용할 수 있게 만든다.

예시 메뉴:
```text
1. 도서 등록
2. 회원 등록
3. 도서 대출
4. 도서 반납
5. 도서 목록 조회
6. 회원 목록 조회
7. 현재 대출 목록 조회
8. 회원별 대출 조회
9. 저장
0. 종료
```

#### 조건
- 잘못된 입력(숫자 대신 문자 등)에 대한 기본 방어 로직을 넣는다.
- 메뉴 루프는 `main`에 과도하게 몰아넣지 않는다.
- 출력 메시지는 사용자가 현재 상태를 이해할 수 있게 작성한다.

### 7. JUnit 테스트 추가
핵심 비즈니스 로직을 테스트로 검증한다. (UI 제외)

최소 테스트 케이스:
- 도서 등록/회원 등록 정상 동작
- 대출 성공
- 이미 대출 중인 책 재대출 시 예외
- 회원 대출 한도 초과 시 예외
- 반납 성공 후 재대출 가능
- 없는 책/회원으로 대출 시 예외
- 현재 대출 목록/회원별 대출 목록 조회 검증

#### 권장
- 테스트 데이터 생성 헬퍼 메서드 사용
- 예외 메시지까지 검증해보기 (`assertThrows` + message)

## 권장 구현 단계 (학습 순서)
한 번에 다 만들지 말고 아래 순서로 진행한다.

1. 도메인 객체(`Book`, `Member`, `Loan`) 먼저 작성
2. 메모리 기반 서비스(`LibraryService`)에서 등록/조회 구현
3. 대출/반납 비즈니스 규칙 + 예외 처리 구현
4. JUnit 테스트로 핵심 로직 먼저 안정화
5. 파일 저장/로드 구현
6. 콘솔 메뉴(UI) 연결
7. 종료/재실행 테스트로 저장 복원 확인

## 권장 패키지 구조 예시
```text
src/main/java/.../
  domain/
    Book.java
    Member.java
    Loan.java
  service/
    LibraryService.java
  exception/
    AlreadyLoanedException.java
    ...
  storage/
    FileStorage.java
    LibraryFileRepository.java
  ui/
    LibraryConsoleApp.java
  Main.java
```

## 규칙
- `null` 반환보다 빈 컬렉션 반환 또는 예외를 우선한다.
- 비즈니스 규칙은 UI가 아니라 서비스에서 검사한다.
- 매직 넘버/문자열은 상수로 관리한다. (예: 기본 대출 한도, 데이터 파일 경로)
- 메서드가 너무 길어지면 역할 기준으로 분리한다.
- 저장 포맷은 단순하게 시작하고, 동작 확인 후 개선한다.

## 오늘의 학습 포인트
#### 반드시 막히는 지점
- "대출 가능 여부" 상태를 `Book`에 둘지, `Loan` 조회로 계산할지?
- `Map`과 `List`를 같이 쓸 때 데이터 정합성을 어떻게 유지할지?
- 예외를 어디서 던지고 어디서 잡아야 책임이 깔끔해지는지?
- 파일 저장 로직이 서비스 로직을 침범하지 않게 어떻게 분리할지?

## 추가 확장 과제 (선택)
- 연체 여부 계산 (`dueDate` 기준)
- 반납 예정일 연장 기능
- 도서 검색 (제목/저자 키워드)
- 저장 포맷 버전 필드 추가
- 관리자/일반 사용자 권한 분리
