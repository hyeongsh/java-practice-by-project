우선 싱글스레드 기반 제대로 예외가 터지는 등을 확인했어. 재현가능한 멀티스레드 부분은 이해를 못하겠어서 조금 더 쉽게 설명해주고, `shutdownRequested` 같은 플래그로 바꾸는 것이 `addCount`를 드는 것과 어떤 차이가 있는건지? 가시성이라는 표현이 정확히 어떤 의미인지? 등을 알아야 할 것 같아.

# 답변

좋아요. 지금 막히는 지점이 정확합니다. 여기 이해되면 프로젝트4 핵심 거의 끝입니다.

## 1) "재현 가능한 멀티스레드 테스트"가 뭐냐 (쉽게)

지금 만든 테스트는 이런 구조예요.

- A 작업: `getAllOrders()`
- B 작업: `addOrder(order4)`

둘을 거의 동시에 실행하고 결과를 보면:

- A가 먼저 끝나면 `3`
- B가 먼저 끝나면 `4`

즉 결과가 스케줄링(실행 순서) 운에 따라 바뀝니다.  
이건 "동시성 검증 테스트"라기보다 "타이밍 복불복 테스트"에 가까워요.

### 재현 가능한 테스트는?

검증 대상을 고정하면 됩니다.

예:

- 스레드 10개가 각각 주문 100개씩 추가
- 모든 작업이 끝날 때까지 기다림 (`Future#get()`, `awaitTermination()`)
- 최종 주문 수는 반드시 1000개
- 특정 고객의 총합도 반드시 기대값

순서는 달라도 최종 결과가 고정되므로 테스트가 재현 가능해집니다.

핵심:

- "중간 상태"를 assert 하지 말고
- "모든 작업 완료 후 최종 상태"를 assert 하기

## 2) 왜 `addCount`는 `volatile` 예시로 부적절하고, `shutdownRequested`는 적절한가?

### `addCount += 1`의 문제

`addCount += 1`은 한 번에 처리되는 것처럼 보이지만 실제로는 대략 아래 단계로 나뉩니다.

1. 값 읽기
2. `+1` 계산
3. 값 다시 쓰기

여러 스레드가 동시에 실행하면 증가가 덮어써질 수 있습니다.  
즉, 원자성(atomicity)이 없습니다.

`volatile`을 붙여도 해결되지 않습니다.

- `volatile`은 "값이 보이게" 해줌 (가시성)
- 하지만 "증가 연산을 한 덩어리로 안전하게 처리"하진 못함 (원자성 X)

그래서 카운터는 보통 다음 중 하나를 사용합니다.

- `synchronized`
- `AtomicInteger`

### `shutdownRequested`가 왜 좋은 예시인가?

예:

```java
private volatile boolean shutdownRequested = false;

public void requestShutdown() {
    shutdownRequested = true;
}

public void workerLoop() {
    while (!shutdownRequested) {
        // 작업
    }
}
```

이 경우는 복합 연산이 아니라 단순히:

- 한 스레드가 `true`를 씀
- 다른 스레드가 그 값을 읽음

이런 "상태 전달"은 `volatile`의 대표적인 사용처입니다.

## 3) "가시성(visibility)"이 정확히 뭐냐?

쉽게 말하면:

- 한 스레드가 값을 바꿨을 때
- 다른 스레드가 그 변경된 값을 제때 볼 수 있느냐

를 의미합니다.

### 가시성 문제 예시 (`volatile` 없음)

스레드 A:

```java
shutdownRequested = true;
```

스레드 B:

```java
while (!shutdownRequested) {
    // 계속 반복
}
```

문제는 B가 최신 값을 바로 안 볼 수 있다는 점입니다.

- 캐시된 값/최적화된 값만 계속 볼 수 있음
- A가 `true`로 바꿔도 B는 계속 `false`처럼 볼 수 있음

`volatile`을 붙이면:

- 쓰기한 값이 다른 스레드에 보이도록 하고
- 읽는 쪽도 최신 값을 다시 보게 함

즉, `volatile`은 "최신 값이 스레드 간에 보이게 하는 장치"입니다.

## 4) 정리 (과제용으로 외우기 좋은 버전)

- `synchronized`
  - 원자성 + 가시성 보장
  - 임계구역(공유 상태 전체 일관성) 보호에 사용
- `volatile`
  - 가시성 보장
  - 복합 연산(`++`, `+=`) 원자성은 보장하지 않음
- `addCount++`
  - `volatile`만으로 안전하지 않음
- `shutdownRequested`
  - 상태 전파(종료 요청) 플래그에 적합

## 5) 네 프로젝트에 적용하면

- `addCount`는 제거하거나 `AtomicInteger`로 변경
- `volatile boolean shutdownRequested` 같은 플래그를 예시로 추가
- 주석/기록에 아래 내용 명시
  - `volatile`은 가시성 보장
  - 복합 연산 원자성은 보장하지 않음
  - 컬렉션 동기화는 `synchronized`가 필요

# 추가 질문

`requestShutdown()`으로 변경한 값을 1초 안에 반영했는지 확인하는 테스트인 거야? 그리고 `volatile` 키워드가 없어도 테스트가 50번 넘게 통과하는데, 무한루프 얘기가 의미가 있는 거야?

# 추가 답변

좋은 질문이고, 이 부분이 `volatile`에서 제일 헷갈리는 지점입니다.

## 1) 지금 테스트는 "속도 테스트"가 아니라 "가시성 테스트"

형태만 보면 맞아요. 겉으로는:

- 메인 스레드가 `requestShutdown()` 호출
- worker 스레드가 1초 안에 종료하는지 확인

처럼 보입니다.

하지만 핵심은 "1초 안에 빠르게 반영되었는가"가 아니라:

- 한 스레드가 바꾼 값(`true`)을
- 다른 스레드가 볼 수 있어서
- 루프를 빠져나오느냐

즉, **가시성(visibility)** 검증입니다.

`1초`는 성능 기준이 아니라 테스트가 무한 대기하지 않게 하는 timeout(안전장치)입니다.

## 2) `volatile` 없어도 왜 테스트가 통과할 수 있나?

이것도 정상입니다.

`volatile`이 없다고 해서 항상 실패하는 게 아닙니다.

실행 환경에 따라:

- 우연히 최신 값이 빨리 보이거나
- JVM/JIT 최적화가 그 실행에서는 문제를 만들지 않거나
- 스케줄링/런타임 타이밍 때문에 결과적으로 값이 관찰되어

테스트가 여러 번 통과할 수 있습니다.

즉:

- `volatile` 없음 = "반드시 실패"가 아님
- `volatile` 없음 = "보장이 없음"이 핵심

## 3) 왜 "캐시가 언젠가 반영될 텐데"가 충분하지 않나?

직관은 맞지만, Java 동시성에서는 캐시 감각만으로 판단하면 부족합니다.

핵심은 Java 메모리 모델(JMM)의 **보장(happens-before)** 입니다.

`volatile`이 없으면:

- 메인 스레드의 쓰기 (`shutdownRequested = true`)
- worker 스레드의 읽기 (`while (!shutdownRequested)`)

사이에 "반드시 이 변경을 봐야 한다"는 규칙이 없습니다.

그래서:

- 볼 수도 있고
- 안 볼 수도 있고
- 언제 볼지도 보장되지 않습니다

## 4) 무한루프 얘기가 왜 나오는가?

무한루프는 "항상 그렇게 된다"는 뜻이 아니라, **그럴 수 있는 코드 구조**라는 뜻입니다.

특히 `volatile`이 없으면 JIT 최적화/재읽기 보장 부재 때문에, worker가 최신 값을 계속 안 보는 상황이 이론적으로 가능하고, Java는 그걸 금지하지 않습니다.

그래서 테스트가 50번 통과해도:

- "이번엔 운 좋게 보였다"일 수는 있어도
- "항상 안전하다"는 증명은 아닙니다

## 5) 중요한 결론: `volatile`은 "재현"보다 "보장"

`volatile`의 가치는:

- 실패를 쉽게 재현하게 해주는 도구라기보다
- JVM/JIT/CPU에게 "이 값은 스레드 간 공유 상태이니 가시성을 보장하라"라고 명시하는 정확성 계약

입니다.

정리하면:

- 테스트 통과 횟수는 참고만 가능
- 안전성 판단 기준은 Java 메모리 모델 규칙
- `shutdownRequested` 같은 플래그에는 `volatile`이 맞음
